# Концепт: навык и фрейм — один эффект, разные исполнения

> [!warning] Не канон
> Это подача идеи для обсуждения. Она не меняет действующие `Registry_Skill_Types`, `AllowedFrames`, теги, мастерство или наборы P/Q/E. Принять её можно только отдельным решением и последующей правкой канонических страниц.

## Тезис

Навык принадлежит `Combo`: он создаёт один известный игроку эффект, с одной ценой батареи, телеграфом, Recovery и набором контрмер. Только явно помеченный frame-translated навык использует текущий допустимый Frame, чтобы определить, **как именно тот же эффект выходит в мир**. Frame не выдаёт и не отнимает навык.

Поэтому нож не является ключом к антихилу. Он может быть лучшим ближним исполнителем антихила: быстрой открытой раной с короткой дистанции. Магострел может передать тот же срыв восстановления на расстоянии, но с иной заметностью, надёжностью попадания и ценой. Оба персонажа владеют одним навыком; их риск и боевая позиция различаются.

```text
Combo -> фиксирует эффект, долг и контрмеры
Frame -> переводит эффект в свою геометрию и экспозицию
Tag   -> открывает новый допустимый Frame или телесную возможность
Mastery -> улучшает исполнение одного выбранного Frame
```

## Зачем это нужно

Существующее обещание [[04_Player_Entities/Proficiency_Arsenal#1. Адаптивный Арсенал (Weapon Gating)|AllowedFrames]] — расширять допустимый арсенал тегами — конфликтует с моделью, где Q/E требует семейство оружия. Если активный навык работает только с одним семейством, новый Frame может сделать уже взятый Q/E практически мёртвым. Тег тогда становится не ростом билда, а обязательным ключом к работоспособности Combo.

Этот концепт сохраняет два желания одновременно:

- фреймы остаются разными и ощущаются физически;
- базовый набор Combo не выключается из-за смены допустимого фрейма;
- тег делает персонажа сильнее через новую тактическую форму, а не через право нажать старую кнопку;
- мастерство не превращается в общий бонус ко всему семейству оружия.

## Инварианты предложения

### 1. Валидность навыка не зависит от семейства фрейма

Для активного навыка нельзя использовать `compatible_frames:: family:<tag>` как условие доступа. Если `current_frame` входит в `AllowedFrames` и навык по природе использует Frame, он получает исполнение через текущий Frame.

Жёсткое предметное требование остаётся допустимым только для процедуры, которая буквально требует внешний аппарат, особый интерфейс или узел мира: например, аномальная процедура через устройство. Это не оружейная специализация Combo.

### 2. Фрейм переводит только явно разрешённый навык

Не все P/Q/E используют Frame. Навык выбирает один из двух режимов:

```text
[execution_authority:: combo_locked | frame_translated]
# default: combo_locked

[frame_inherits:: none | kernel, delivery_form]
# обязателен только при frame_translated
```

`combo_locked` сохраняет собственные `kernel`, `delivery_form` и форму исполнения. Аура, поле, процедура, барьер, self-buff и действие телом/устройством не получают скрытый профиль от оружия.

`frame_translated` фиксирует эффект и тактическую цель у Combo, а у текущего Frame берёт только `kernel` и `delivery_form`. Это режим для действия вида «создать известный эффект на одном враждебном подтверждённом попадании».

### 3. Эффект не меняется при смене Frame

Frame не имеет права подменить:

- `effect_domain`, статус/исход, окно и `target_scope` навыка;
- батарейную цену и общий долг;
- ключевую контригру;
- идентичность Q/E в интерфейсе игрока.

У `frame_translated` навыка он может менять только унаследованные `kernel` и `delivery_form`, а также параметры их исполнения: дистанцию, геометрию, требование подтверждённого попадания, сенсорный tell, Exposure и надёжность наложения. Это одна способность, а не скрытые альтернативные Q/E.

Длительность windup или Recovery может различаться лишь внутри заранее объявленного коридора. Frame не может убрать обязательную фазу, превратить мгновенное действие в `maintained`, сменить `effect_persistence`, `interrupt_rule`, тип контригры, `target_scope` или стоимость батареи.

### 4. Специфическая причинность — бонус формы, а не запрет доступа

Лезвие, укол, удар и выстрел не обязаны одинаково объяснять эффект. Они обязаны честно его показывать. Если нужный эффект не может иметь правдоподобного исполнения через данный Frame, это сигнал, что эффект сформулирован слишком узко для Combo.

Например, «разрезать сосуд и сорвать лечение» — техника клинка. «Подавить входящее восстановление после враждебного попадания» — эффект Combo, который лезвие выражает раной, удар — травмой, а магострел — нарушенным импульсом в точке попадания.

### 5. Узкая техника возможна, но не живёт в P/Q/E Combo

Если автору нужен именно эффект, честный только для одного типа оружия, он относится к технике или мастерству конкретного Frame. Он не может быть обязательной частью базового Q/E, Gate Check или единственным ответом на системную угрозу.

## Модель исполнения

У `combo_locked` навыка есть только собственный контракт Combo. `frame_translated` навык описывается дважды: его неизменяемое ядро и изменяемое исполнение.

```text
skill_core = {
  effect,
  target rule,
  energy cost,
  telegraph,
  recovery,
  counterplay
}

frame_execution = {
  delivery geometry,
  application condition,
  exposure,
  reliability,
  frame-specific sensory tell
}
```

При выпуске навыка:

```text
if skill requires a Frame:
    require current_frame in AllowedFrames
    if execution_authority == frame_translated:
        inherit kernel and delivery_form from current_frame.execution_profile
    else:
        resolve the combo-owned contract unchanged
else:
    resolve through body, device or environment node
```

`execution_profile` — не новый источник силы и не отдельный комплект навыков. Это словарь того, как уже разрешённый Frame доставляет известный эффект. Он применяется только к `frame_translated`.

## Различимость Combo и контролируемый смысл

`semantic_intent` не может быть свободной авторской строкой. Для `frame_translated` навыка он обязан ссылаться на контролируемый `intent_id` из будущего `Registry_Semantic_Intents`.

```text
[semantic_intent:: none | intent_id]
[status_effect:: none | effect_id]
[outcome_rule:: none | rule_id]
```

`status_effect` и `outcome_rule` взаимно заменяют друг друга только там, где это допускает конкретный intent: урон, защита или перемещение могут завершаться правилом без статуса. Оба поля должны быть реестровыми ID, а не свободным текстом.

Предлагаемый минимум записи `intent_id`:

```text
intent_id: disrupt_restore_on_hostile_hit
meaning: создать окно, в котором входящее восстановление цели ослаблено
allowed_outcomes: healing_suppression | restoration_fracture
target_policy: single_hostile
forbidden_changes: area, maintained, heal_denial
```

Проверка §12 получает два режима:

```text
if execution_authority == combo_locked:
  identity = (kernel, delivery_form, target_scope, effect_domain)

if execution_authority == frame_translated:
  identity = (
    semantic_intent,
    effect_domain,
    target_scope,
    primary_window_function,
    status_effect | outcome_rule
  )
```

Две раздельные формулировки вроде `disrupt_restore_on_hostile_hit` и `suppress_recovery_on_enemy_strike` не смогут обойти проверку: автор обязан выбрать существующий `intent_id` либо зарегистрировать реально новый смысл с другими границами.

## Пример: один антихил, три честные формы

В примере статусом остаётся [[05_Combat_Survival/_Registries/Registry_StatusEffects#Подавление восстановления|`healing_suppression`]]. Никакой Frame не создаёт другой статус и не запрещает лечение полностью.

| Текущий Frame | Видимое исполнение | Условие приложения | Что он платит |
|---|---|---|---|
| Короткий клинок | открытая рана и сорванный восстановительный ритм | контакт по уязвимой части тела | ближняя дистанция и опасная Exposure |
| Проломный ударник | травматический сбой после тяжёлого удара | подтверждённый сильный impact | медленный телеграф и Recovery |
| Магострел | нарушенный импульс в точке попадания | точное попадание | заметная линия, расход и контр-давление на стрелка |

Все три исполнения сохраняют `target_scope:: single`, `effect_persistence:: attached`, единый статус и одинаковую структуру долга. Здесь нож может быть лучшей формой для короткой засады, а магострел — лучшей формой давления на лекаря из-за укрытия. Ни один не становится «единственным правильным антихилом».

## Теги и мастерство

### Теги

Тег может:

- добавить Frame в `AllowedFrames`;
- снять физиологический запрет на его использование;
- дать телесную возможность, нужную для его удержания;
- изменить цену или риск этого расширения: шум, Диссонанс, уязвимость, потеря другой ёмкости.

Тег не может:

- сделать существующий Q/E доступным впервые;
- заменить конкретный Frame в качестве единственного валидного;
- дать эффект Combo, бесплатный импульс или второй источник уже оплаченной силы.

Теговая сборка потому сильнее не числом «разрешённых кнопок», а диапазоном позиций: тяжёлый Frame открывает пролом и удержание угла, лёгкий — скрытную доставку и выход, дальний — давление через пространство.

### Мастерство

Мастерство относится к **экземпляру/доктрине конкретного Frame**, не к широкому `blade`, `blunt` или `arcanegun`. Оно может улучшить его `execution_profile`: стабильность, окно отмены, контроль Exposure, точность или дополнительное читаемое окно исполнения.

Мастерство не должно:

- увеличивать базовую силу эффекта Combo;
- делать другой Frame невалидным;
- автоматически распространяться на весь тег семейства;
- совмещаться в одном теге с выдачей единственного нужного Frame и его готового мастерства.

## Что отвергает концепт

| Модель | Почему не подходит |
|---|---|
| `family:edged` как допуск Q/E | При текущем ростере это почти прямое требование одного ножа; теги становятся налогом на рабочий навык. |
| `projectile` без цены формы | Все дальние фреймы формально валидны, но один с лучшими базовыми параметрами вытеснит остальные. |
| Один визуальный язык для всех фреймов | Стирает причинность оружия: молот, нож и магострел ощущаются одной кнопкой. |
| Клинковый эффект как обязательный Q/E | Либо врёт о причинности других Frame, либо делает Combo нерабочей вне клинка. |

## Проверка перед возможной канонизацией

Для каждого Q/E нужно подтвердить:

- есть ли у него честная форма без Frame; если нет — есть ли честное исполнение у каждого базово допустимого Frame;
- имеет ли навык явный `execution_authority`, а `frame_translated` — только разрешённое наследование `kernel, delivery_form`;
- не меняет ли Frame статус/исход, `target_scope`, контригру или структуру долга, а только доставку и коридор длительности фаз;
- существует ли `semantic_intent` в контролируемом реестре и не является ли он новым именем для уже существующего смысла;
- не стал ли новый тег единственным способом запустить старый Q/E;
- есть ли у каждого нового Frame собственная выгода и собственная Exposure;
- не является ли узкая техника единственным ответом на Gate Check, лечение, выход или Аномалию;
- понимает ли игрок по телеграфу, какой именно Frame-формой в него играют.

## Открытые решения

1. Какой формат и владелец будущего `Registry_Semantic_Intents`: самостоятельный реестр или раздел реестра грамматики навыков?
2. Как измерять допустимый коридор длительности фаз без превращения Frame в источник отдельного эффекта?
3. Что именно считается объектом мастерства: класс Frame, конкретный registry-ID или закреплённая доктрина?
4. Какие стартовые Q/E намеренно остаются узкими техниками фрейма, а не ядром Combo?
