# Аудит MVP-матрицы 3×3

> Цель аудита — отделить готовность проектного контракта от готовности девяти ячеек. На текущем этапе каркас согласован, но восемь ячеек не имеют утверждённого содержания, а у Белки × Авангарда утверждена только пассивная основа.

## Проход A: соответствие утверждённым решениям

| Критерий | Статус | Где закреплено |
|---|---|---|
| MVP-состав: Ёж/Крыса/Белка × Авангард/Технократ/Странник | проходит | контракт, три реестра |
| Старые curated-комбинации не считаются каноном | проходит | `Registry_Combos` очищен до девяти слотов |
| Все `P/Q/E` являются смешением расы и специализации | проходит на уровне контракта | `MVP_3x3_Design_Contract`, `Ability_Synergy`, `Shell_Construction` |
| Жаба, Ящерица, Страж и Догмат сохранены как expansion-основы | проходит | `Registry_Races`, `Registry_Specs` |
| Ячейка — шасси архетипа, а не фиксированный билд | проходит | контракт |
| У каждой готовой ячейки будет несколько оружейных и модульных доктрин | проходит на уровне контракта | контракт, `Proficiency_Arsenal`, `Thermos_System` |
| Синергия строится по совместимости окон и состояний | проходит | контракт, `Ability_Synergy` |
| Условия Аномалии и мобы описываются признаками | проходит | контракт |
| Соло валиден, но не универсален | проходит | контракт |
| Повторы классов, рас и точных ячеек разрешены | проходит | контракт |
| Повтор усиливает общий план и общие слабости без искусственного штрафа | проходит | контракт |
| Мобильное тело сочетается с медленными, уязвимыми действиями | проходит | контракт и Dark and Darker reference |
| Странник остаётся наиболее маршрутно-мобильной методологией | проходит | `Registry_Combos`, `Registry_Specs` |
| Авангард сохраняет уязвимости | проходит | таблица Двойного Парадокса |
| Базовая слабость не исчезает после `prof:: 3+` | проходит | контракт, `Combat_Profile_Pipeline`, `Two_Paradox_Vector_Matrix` |
| Белка × Авангард играет от значимых импульсов движения | проходит | контракт, реестры рас и комбо |
| Снайперское чтение Белки не стало главным каноном | проходит | `Registry_Combos` |
| Технические рекомендации отделены от игрового канона | проходит | technical reference и карта переноса |

## Проход B: критический слой каркаса

| Риск | Состояние после слияния |
|---|---|
| Одна кнопка безопасно входит, побеждает и выходит | запрещено контрактом; проверяется на каждой будущей `Q/E` |
| Бесплатный ресурс или бесконечная перегрузка | запрещено контрактом; особенно важно для Белки × Технократа |
| Иммунитет вместо контригры | удалён из биологических основ; запрещён контрактом |
| Вечный контроль и повторный stagger-lock | предусмотрены diminishing returns/окно устойчивости |
| «Если Порт, то обязательно Жаба» | заменено на теги условий и профили угроз |
| Один дешёвый комплект сохраняет всю силу | введены loadout stress tests |
| Один дорогой Frame обслуживает всю бюджетную группу | введена отдельная проверка |
| Одинаковый состав штрафуется системой | искусственный штраф запрещён |
| Союзник впервые включает функциональность способности | запрещено правилом соло-подготовки |
| Технический стек выдаётся за доказанный дизайн | разделено на прототип, vertical slice и открытые решения |

## Что ещё не проходит

Ни одна ячейка пока не имеет полного статуса `design_status:: approved`.

Для каждой из девяти ещё нужны:

- фантазия и повторяемый цикл;
- смешанные `P/Q/E`;
- условие, телеграф, цена отмены и Recovery каждой `Q/E`;
- 2–4 доктрины с оружием, Термосом, модулями, батареями и расходниками;
- окна совместимости;
- преимущества и недостатки против условий и профилей мобов;
- соло-проверка;
- проверка двух и трёх копий;
- результаты успеха, отхода и провала;
- числовой прототип и loadout stress tests.

У `squirrel_assault` утверждена только пассивная основа, поэтому её статус остаётся `foundation_approved`, а не `approved`.

## Правило следующего шага

Проектировать по одной ячейке и после каждого черновика проходить:

1. дизайн-проверку контракта;
2. критическую проверку на доминирующую стратегию;
3. проверку проживаемого игрового опыта;
4. проверку арсенала и прогрессии;
5. соло, два повтора и три повтора;
6. условия Аномалии и минимум два профиля мобов.

Только после этого менять `design_status`.
