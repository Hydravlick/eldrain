# Система линий мутаций Аномалии

## Ответственность

Этот документ определяет универсальную модель развивающихся модификаторов Аномалии: выбор линии, переход T1→T2→T3, обязательные данные стадии и требования к читаемости. Конкретный сеттинг, его существа, погода и народные названия принадлежат экземпляру сектора.

## Четыре слоя инстанса

1. **Сеттинг Аномалии** задаёт погибший мир, катастрофу, визуальную грамматику и неизменные причинные законы.
2. **Базовая экосистема** задаёт обычный пул мобов и POI локации.
3. **Доминирующая линия мутации** добавляет условные формы, варианты поведения, ловушки, состояния префабов и погодное выражение.
4. **Tier-стадия** определяет текущее проявление выбранной линии.

В стартовом контракте инстанс использует одну доминирующую развивающуюся линию. Базовая экосистема сохраняется на всех стадиях.

## Выбор и переход стадии

1. Генератор выбирает полную линию при создании инстанса и сохраняет её в `Generation Snapshot`.
2. T1 размещает раннюю форму и наследуемые признаки.
3. Первый Phase Shift активирует T2-стадию той же линии.
4. Второй Phase Shift активирует T3-стадию и её предельное выражение.
5. Действия игроков меняют локальные префабы, сигналы, маршруты, добычу и состояние встреч. Выбранная линия остаётся частью seed текущего инстанса.

## Контракт тегов

| Поле | Назначение |
|:---|:---|
| `anomaly_family` | сеттинг и его причинный закон |
| `mutation_line` | выбранная законченная линия |
| `mutation_stage` | конкретная стадия линии |
| `tier_expression` | T1, T2 или T3 |
| `inherited_tells` | признаки, сохраняющиеся между стадиями |
| `mob_additions` | условные мобы поверх базового пула |
| `mob_variants` | разрешённые изменения базовых мобов |
| `base_ecology_effect` | изменение взаимодействий базовой экосистемы |
| `prefab_state` | подготовленное состояние дверей, ловушек, спавнеров или маршрутов |
| `weather_state` | погодное выражение стадии |
| `study_output` | знание, которое способен подтвердить игрок |
| `production_class` | `state`, `variant` или `set_piece` |

Структурированные значения хранятся в [[08_World_Generation/_Registries/Registry_Anomaly_Mutations|Registry_Anomaly_Mutations]]. Экземпляр сектора отображает связанные записи через Dataview либо DataviewJS.

## Контракт стадии

Каждая стадия содержит:

- визуальный, звуковой и поведенческий признак;
- наследуемую связь с соседними стадиями;
- изменение поведения, пространства или восприятия помимо числового усиления;
- решение `исследовать / использовать / обойти / отступить`;
- мягкую контрмеру;
- исследовательский или добывающий результат;
- реализуемый производственный класс.

## Информационная кривая

| Tier | Доступное знание |
|:---|:---|
| T1 | распознать семейство признаков и получить ранний прогноз |
| T2 | подтвердить выбранную линию и увидеть зрелую причинность |
| T3 | прочитать текущее правило, использовать его ритм и получить зрелый образец |

Каждый Tier содержит собственный физический сигнал до первого необратимого обязательства. UI подтверждает наблюдаемую причинность.

## Граница данных и представления

- системная заметка хранит универсальные правила;
- заметка экземпляра хранит сеттинг и его законченные линии;
- реестр линий хранит ID и структурированные стадии;
- реестр мобов хранит определения существ и вариантов;
- манифест сектора хранит пространственный контракт и встраивает экземпляр;
- Dataview-представления собирают списки из реестров вместо копирования определений.

## Честность для пермадеза

- Наследуемый признак появляется раньше смертельного выражения.
- Дальний урон, ловушка и фазовая смена имеют воспринимаемый предвестник.
- Обязательный маршрут сохраняет телеграфируемую альтернативу либо момент отступления.
- Контрмера создаёт преимущество и цену, а не полную иммунность.
- Вызванный ответ использует конечный локальный ресурс и не образует бесконечный фарм.

## Производственные границы

- Пространственные изменения используют подготовленные состояния модулей, NavLink, материалы и hazard volume.
- Условными формами становятся размеченные ассеты.
- Варианты мобов выбираются из таблицы совместимости.
- Погода управляет подготовленными состояниями и читаемыми импульсами.
- Set-piece имеет узкий триггер и отдельную проверку прототипа.

## Проверки прототипа

- игрок распознаёт линию по раннему признаку;
- следующая стадия воспринимается как развитие знакомого правила;
- контрмера используется как выбор, а не обязательный иммунитет;
- поздний вход получает достаточный локальный сигнал;
- причина потери и путь отказа остаются понятными.
